# 🛡️ 경남외고 시스템 보안 관리 지침 (SECURITY POLICY)

본 지침은 시스템 관리자 전용 기능의 오용을 방지하고, 비관리자에게 민감한 정보나 제어 도구가 노출되지 않도록 하기 위한 **Antigravity AI의 절대 준수 원칙**입니다.

## 1. UI 노출 제어 원칙 (View Layer)
- **원칙**: 관리자 전용 버튼, 메뉴, 정보 패널은 비관리자 계정(`isAdmin === false`)에서 아예 렌더링되지 않아야 한다.
- **실행**: 
    - 단순히 CSS로 숨기지 말고, `{isAdmin && ( <Component /> )}`와 같이 조건부 렌더링을 사용하여 DOM 수준에서 완전히 배제한다.
    - 호버(Hover) 시 나타나는 버튼이나 툴팁도 예외 없이 동일한 규칙을 적용한다.

## 2. 함수 실행 차단 원칙 (Logic Layer)
- **원칙**: UI에서 버튼이 숨겨졌더라도, 브라우저 콘솔 등을 통한 직접적인 함수 호출을 방어하기 위해 모든 관리자 전용 함수의 시작점에는 권한 검증 로직이 있어야 한다.
- **실행**:
    - `handleSave`, `handleDelete`, `handleClone` 등 데이터 변이(Mutation)를 일으키는 함수의 첫 행에 `if (!isAdmin) return;`을 삽입한다.
    - 필요 시 경고 창(`alert`)을 통해 비정상적인 접근 시도임을 사용자에게 명시한다.

## 3. 데이터 요청 필터링 원칙 (Data Layer)
- **원칙**: 클라이언트 측 필터링(JS filter)에 의존하지 않고, 서버에서 데이터를 가져올 때(Fetch)부터 소유권이나 공개 여부를 확인하여 응답에 포함시킨다.
- **실행**:
    - Supabase RLS(Row Level Security)를 기반으로 하되, 클라이언트 쿼리에서도 `isAdmin`이 아닌 경우 필수 필터를 강제한다.

## 4. 작업 및 검증 절차
- 관리자 권한 관련 기능을 수정할 때는 반드시 다음 단계를 거친다.
    1. 수정 전 **백업 코드** 확보.
    2. 수정 후 **비관리자 페르소나**로 해당 기능이 보이지 않고 작동하지 않는지 교차 검증.
    3. 본 `SECURITY_POLICY.md` 지침에 위배되는 사항이 없는지 최종 체크.

---
*본 지침은 사용자님의 승인 하에 수립되었으며, Antigravity AI는 향후 모든 작업에서 이 지침을 최우선 순위로 준수할 것을 약속합니다.*
